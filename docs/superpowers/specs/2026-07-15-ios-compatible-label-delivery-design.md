# macOS 与 iOS 兼容的 IP 标签交付设计

## 目标

在不修改上游机场订阅或现有 macOS 本地镜像机制的前提下，让 Surge iOS 也能读取同一份已扫描完成的 IP 标签节点列表。扫描继续由 Sub-Store 每 12 小时执行一次；iOS 只下载快照，绝不触发 IP 扫描。

## 已有状态

- Sub-Store Script Operator 负责读取原始订阅、经节点取得出口 IP、添加 IP 标签，并在整轮完成后上传 Worker 快照。
- macOS 从 Worker 拉取快照，先校验，再原子写入本地 `ip-labels.conf`；这条路径保持不变。
- 远程 `policy-path` 在现有环境曾有超时及格式兼容问题，因此 iOS 不能直接复用旧的通用订阅接口，也不能替代 macOS 的本地文件镜像。

## 推荐架构

### 1. 单次扫描，共享快照

Sub-Store 每 12 小时完成一次扫描后，向现有 Worker 上传一个完整快照。Worker 保存当前快照与更新时间；macOS 和 iOS 均从该快照读取，因此两端显示的 IP、普通评分、原生性、住宅/机房、人机流量和 GPT 评分一致。

### 2. 保留 macOS 本地镜像

macOS 继续通过 Keychain 中的读取令牌下载私有快照，使用 `surge-cli --check` 验证，并原子替换 `ip-labels.conf`。请求或校验失败时旧文件保留，避免影响现有选择组。

### 3. 新增 iOS 专用策略资源

Worker 新增一个仅用于 Surge `policy-path` 的只读端点。端点：

- 只返回已完成的静态快照，禁止在请求路径中扫描、查询 IP 或访问 Sub-Store；
- 使用适合 `policy-path` 的纯策略条目格式和明确的 `Content-Type: text/plain; charset=utf-8`；
- 以独立的 iOS 读取令牌鉴权，可独立轮换或撤销；
- 对成功快照设置短时私有缓存提示，但不公开缓存、不会记录令牌。

单独令牌的作用是：iPhone 中的模块参数不可避免由设备持有；泄露时只撤销 iOS 令牌，不影响 macOS 本地镜像。

### 4. iOS 先使用隔离测试策略组

提供可 URL 安装的 iOS 模块。模块只新增一个独立策略组，例如 `🧪 IP 标签 iOS`，并将其 `policy-path` 指向 iOS 专用端点。它不替换现有的“手动切换”、自动选优或任何业务策略组。

用户在 iOS 上验证以下内容后，才手动选择是否把正式手动切换组改为此资源：

1. 外部资源更新状态为就绪；
2. 节点数量和名称与 Sub-Store 预览基本一致；
3. 节点可进行延迟测试且标签显示正常；
4. 远程资源临时失败时，已加载的上一版本仍可使用。

## 更新节奏

- 扫描：每 12 小时一次，仍由 Sub-Store 执行。
- 快照上传：仅在每轮扫描完成后执行一次。
- macOS 本地镜像：沿用扫描后约 25 分钟的安全同步，并可手动运行。
- iOS 外部资源：每小时拉取一次静态快照；它不增加 IP 查询次数。扫描完成后，iOS 最迟于下一次外部资源更新获取结果。

## 错误处理与安全边界

- iOS 端点无快照时返回明确的非成功状态，不返回伪造的空配置。
- 快照格式验证失败时，Worker 不替换已验证的当前快照。
- iOS 模块初始仅新增测试组，失败可直接禁用模块；不影响原订阅和当前网络。
- 不把机场订阅 URL、节点凭据、`SYNC_TOKEN`、macOS 读取令牌或 iOS 读取令牌写入仓库、日志或设计文档。
- iOS 令牌只以用户在模块参数中本地填写的方式使用；公开模块文件只包含占位符。

## 测试与验收

1. Worker 单元测试覆盖 iOS 端点：无效/缺失令牌、无快照、有效快照、响应头和策略正文。
2. 使用 `surge-cli --check` 验证 iOS 端点快照可作为最小策略组资源使用。
3. 在不改动既有策略组的前提下，安装测试模块并确认 macOS 与 iOS 均可加载隔离组。
4. 人为模拟 Worker 请求失败，确认 macOS 保留旧本地文件，iOS 仍保留最后一次成功加载的资源。
5. 确认 12 小时扫描后，上传一次新快照；iOS 的定时下载不会产生额外 IP 扫描请求。

## 不采用的方案

- **让 iOS 读取 Mac 的 `ip-labels.conf`：** iOS 沙盒不能可靠访问 Mac 的 Surge 配置目录。
- **把 macOS 改为完全远程加载：** 会丢失已验证的本地校验、原子替换和故障保留能力。
- **iOS 在打开节点列表时自行扫描：** 会造成延迟、重复查询和数据源压力，也无法保证完整性。
